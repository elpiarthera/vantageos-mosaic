## 5. Three new CI gates (per Standard §18.6 promise)

### Gate 1: Cross-runtime build parity
**Goal**: `tsup` produces ESM+CJS+DTS for react pass AND ESM+DTS for preact pass without errors.
**Location**: `.github/workflows/ci.yml` — new job `build-parity-cross-runtime` after `build` step.
**Implementation**: shell script under `scripts/verify-build-parity.sh` checks `packages/mosaic/dist/react/*.{js,cjs,d.ts}` AND `packages/mosaic/dist/preact/*.{js,d.ts}` existence + non-empty. Fails CI if any miss.
**Failure mode**: missing dist file → CI red with explicit "missing react/<cat> ESM" or "missing preact/<cat> DTS" message.
**Override marker**: `// allow-build-parity-skip: <reason>` in PR description (block hooks read it).

### Gate 2: PeerDeps optionality smoke
**Goal**: `npm pack --dry-run` outputs MUST NOT list `react` or `preact` in installed deps for either runtime consumer scenario.
**Location**: new folder `e2e/peer-resolution/` with 2 minimal consumer fixtures:
- `e2e/peer-resolution/react-only/` — `package.json` declares `react` only, attempts to import `@vantageos/mosaic/preact/progress` → MUST fail at install or import time (peer mismatch caught)
- `e2e/peer-resolution/preact-only/` — `package.json` declares `preact` only, attempts to import `@vantageos/mosaic/react/progress` → MUST fail similarly
**Implementation**: GitHub Actions job runs `npm install --no-package-lock --prefix e2e/peer-resolution/react-only` then `node -e "require('@vantageos/mosaic/preact/progress')"` and asserts exit code matches expected failure.
**Failure mode**: if either scenario passes (peers not strictly checked) → CI red. If either fails for unexpected reason (build error vs peer error) → CI red with diagnostic.
**Override marker**: `// allow-peer-resolution-skip: <reason>` for hotfix windows.

### Gate 3: Coherence dedicated gate
**Goal**: elevate existing `coherence.test.ts` (mosaic-tokens) from generic vitest run to **dedicated** GitHub Actions step that gates merge.
**Location**: `.github/workflows/ci.yml` — new job `coherence-test-mosaic-tokens` after `test` step, OR `coherence-test` as a required status check on the repo.
**Implementation**: `pnpm --filter @vantageos/mosaic-tokens test --reporter=verbose --bail` — bail on first failure for clean report. Job marked as required in branch protection settings.
**Failure mode**: any of the 10 coherence tests failing → CI red + merge blocked even if other tests pass.
**Override marker**: NONE — coherence is the ship-blocker invariant per Standard v1.1 §18.6 (no override allowed).

## 6. Batching strategy proposal

### Option A (recommended): Per-component PRs (1 component = 1 PR)
- 1 PR = React variant + Preact variant + schema + logic + tests + Storybook + registry.yaml entry + coherence test update (if applicable)
- File touch range per PR: ~10-15 files (5-7 per runtime + tests + meta)
- Review burden: lower per-PR — Eta can audit single component end-to-end in one pass
- Velocity: parallelizable — Chi/Hermes/Demeter components don't conflict
- Merge friction: low — stacked PRs avoided since each is independent

### Option B: Per-category PRs (e.g. all input/* in 1 PR)
- 1 PR = all components of one category × 2 runtimes
- File touch range per PR: ~30-50 files
- Review burden: higher per-PR — Eta must audit N components in one PR (axes × N components)
- Velocity: slower — categories ship sequentially, blocking parallel BU consumption
- Merge friction: higher — single revision request blocks entire category

**Recommendation**: Option A. Aligns with v0.2.0 small-PR cadence that worked well Day 90 (5 PRs in one session).
